# Chain Analysis — 연계 분석 (Phase 2)

Phase 1에서 발견된 개별 취약점 후보를 입력받아, 후보 간 연계 관계를 분석하고 실제 공격 시나리오를 도출한다.

## 목적

Phase 1 스캐너는 각각의 취약점 유형(XSS, SSRF, IDOR 등)을 독립적으로 분석한다. 그러나 실제 공격은 여러 취약점을 조합하여 성립한다. 연계 분석은 **이미 발견된 후보들을 연결하여** 완전한 공격 시나리오를 구성한다.

## 입력

메인 에이전트가 연계 분석 에이전트에 전달하는 데이터:

1. **후보 마스터 목록**: Phase 1에서 발견된 모든 후보 (ID, 제목, 유형, 위치, Source→Sink 경로, 실제 URL 경로)
2. **프로젝트 컨텍스트**: 기술 스택, 서비스 아키텍처, 인증 방식, 네트워크 구조
3. **이상 없음 요약**: Phase 1에서 안전으로 판정된 항목의 요약 (참고용)

## 분석 프로세스

### Step 1: 전제조건 매트릭스 작성

각 후보에 대해 **트리거 전제조건**을 분석한다:

```
후보 ID: XSS-2
트리거 전제조건:
  1. [권한] 이용기관 자격으로 전자문서 발송 API 호출 가능
  2. [데이터] 수신자의 전화번호, 이름, 생년월일
  3. [네트워크] drowa-api 엔드포인트 접근 가능
```

모든 후보에 대해 전제조건을 나열한 뒤, **전제조건 유형별로 그룹화**한다:
- `[권한]`: 특정 API/서비스에 대한 인증/인가
- `[데이터]`: 특정 식별자(envelopeId, userId 등) 또는 개인정보
- `[네트워크]`: 특정 도메인/포트 접근
- `[환경]`: 특정 환경(sandbox/production) 한정 여부

### Step 2: 연계 관계 분석

후보 간 **"A가 B의 전제조건을 충족하는가?"** 관계를 분석한다.

분석 질문:
- 후보 A를 악용하면, 후보 B의 전제조건이 충족되는가?
- 후보 A의 출력(탈취된 데이터, 획득한 권한 등)이 후보 B의 입력이 되는가?
- 여러 후보를 순서대로 실행하면 단일 후보로는 불가능한 영향을 만들 수 있는가?

**연계 매트릭스 형식:**

```
         | XSS-1 | XSS-2 | HOST-1 | HOST-2 | ... |
---------|-------|-------|--------|--------|-----|
XSS-1    |   -   |       |        |        |     |
XSS-2    |       |   -   |        |        |     |
HOST-1   |       |       |   -    |        |     |
HOST-2   |       |       |        |   -    |     |
...      |       |       |        |        |  -  |
```

셀에 들어갈 값:
- `→`: 행(A)이 열(B)의 전제조건을 충족함
- `↔`: 양방향 연계
- `+`: 조합 시 영향 증폭
- (빈칸): 연계 없음

### Step 3: 공격 체인 구성

연계 매트릭스에서 도출된 관계를 기반으로, 시작점에서 최종 영향까지의 완전한 공격 시나리오를 구성한다.

**공격 체인 형식:**

```
## 공격 체인 #1: [체인 제목]

공격자 프로필: [누가 이 공격을 수행할 수 있는가]
최종 영향: [공격 성공 시 결과]

Step 1: [후보 ID] — [설명]
  → [후보 A를 악용하여 얻는 것]

Step 2: [후보 ID] — [설명]
  → [Step 1의 결과를 이용하여 후보 B를 트리거]

Step 3: [후보 ID 또는 최종 영향] — [설명]
  → [최종 영향]

관련 후보: [사용된 Phase 1 후보 ID 목록]
```

**체인 분류:**
- **완전 체인**: 모든 단계가 Phase 1 후보로 구성됨. 동적 테스트에서 end-to-end 검증 가능.
- **부분 체인**: 일부 단계에 외부 전제조건(예: 내부망 접근)이 포함됨. 해당 전제조건 하에서 검증 가능.

### Step 4: 독립 후보 정리

어떤 체인에도 포함되지 않은 후보를 별도로 정리한다. 후보별 체인 미구성 사유만 기재한다. 단독 위험도는 각 취약점 상세 섹션에 이미 기술되어 있으므로 여기서 중복하지 않는다.

```
| 후보 | 체인 미구성 사유 |
|------|----------------|
| [후보 ID] | [이 후보가 다른 후보와 연계되지 않는 구체적 이유] |
```

## 출력

연계 분석 에이전트는 다음을 반환한다:

1. **전제조건 매트릭스**: 후보별 전제조건 테이블
2. **연계 매트릭스**: 후보 간 연계 관계 테이블
3. **공격 체인 목록**: 완전한 공격 시나리오 (Step-by-step). 체인이 없으면 "없음".
4. **독립 후보 테이블**: 후보별 체인 미구성 사유 (단독 위험도 제외)
5. **Phase 3 테스트 목표**: 동적 테스트에서 검증할 체인/항목 목록 (체인 단위 우선)

## 에이전트 실행 지침

### 분석 원칙

1. **발견된 후보만 사용한다.** 새로운 취약점을 탐색하지 않는다. Phase 1 결과를 입력으로만 사용한다.
2. **코드 근거를 명시한다.** 연계 관계를 주장할 때, 해당 데이터 흐름의 코드 경로(파일경로:라인번호)를 제시한다. 데이터 흐름을 확인하기 위해 코드를 Read하는 것은 허용한다.
3. **추측 체인을 만들지 않는다.** "A의 출력이 B의 입력이 될 수 있다"는 코드 레벨에서 증명 가능해야 한다.
4. **공격자 프로필을 명확히 한다.** 각 체인이 누구에 의해 실행 가능한지(외부 공격자, 내부자, 이용기관 등) 명시한다.

### 주의사항

- Phase 1에서 "안전"으로 판정된 항목을 재분석하지 않는다.
- 체인이 하나도 구성되지 않을 수 있다. 그 경우 "연계 관계 없음"으로 결론짓고 독립 후보 목록만 반환한다.
