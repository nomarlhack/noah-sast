> ## 핵심 원칙: "다른 사용자의 리소스에 접근할 수 없으면 취약점이 아니다"
>
> 소스코드에서 `findById(id)`가 있다고 바로 IDOR로 보고하지 않는다. 실제로 사용자 A의 인증 정보로 사용자 B의 리소스 식별자를 전송했을 때, 서버가 사용자 B의 리소스를 반환하는 것을 확인해야 취약점이다.
>

### Phase 1: 정찰 (소스코드 분석)

객체 식별자를 받아 리소스에 접근하는 엔드포인트에서 소유권/권한 검증이 누락된 곳을 식별한다.

1. **프로젝트 스택 파악**: 프레임워크/언어 확인 및 인가 메커니즘 파악
   - 인증 방식 (세션, JWT, API 키 등)
   - 인가 프레임워크 (Spring Security, Django Permissions, RBAC, ABAC 등)
   - 글로벌 인가 필터/미들웨어 존재 여부
   - 멀티테넌트 구조 여부 (조직/계정 단위 격리)

2. **엔드포인트 식별**: 객체 식별자를 파라미터로 받는 엔드포인트를 찾는다
   - 경로 파라미터로 ID를 받는 엔드포인트 (예: `/users/{id}`, `/orders/{orderId}`)
   - 쿼리/바디 파라미터로 ID를 받는 엔드포인트
   - 내부용(internal) API 경로가 외부에 노출되는지 확인

3. **소유권 검증 로직 추적**: 각 엔드포인트에 대해 식별자와 인증된 사용자 간의 관계 검증 여부를 확인
   - 데이터 접근 시 인증된 사용자의 ID/계정으로 스코핑하는지 (예: `WHERE owner_id = currentUser AND id = ?`)
   - 미들웨어/필터 레벨에서 리소스 소유권을 검증하는지
   - 서비스 레이어에서 접근 권한을 확인하는지
   - 아무런 소유권 검증 없이 식별자만으로 객체를 조회하여 반환하는지

4. **접근 제어 설정 분석**: Security 설정에서 경로별 인가 규칙 확인
   - 인증 없이 접근 가능한 경로 (permitAll, anonymous 등)
   - 역할 기반 접근 제어만 적용되고 객체 수준 권한 확인이 없는 경로
   - 내부 API 경로에 대한 인증/인가 설정

5. **후보 목록 작성**: 식별자를 받아 객체에 접근하되 소유권 검증이 없는 엔드포인트를 정리. 데이터 흐름 추적을 완료한 뒤에도 공격 경로가 없으면 버린다. 추적 없이 직관으로 버리지 않는다.

#### IDOR가 아닌 패턴 (오탐 주의)

- **공개 리소스**: 설계상 누구나 접근 가능한 리소스 (공개 게시글, 공개 프로필 등). 비즈니스 요구사항을 확인한다.
- **인증된 사용자 자신의 리소스만 반환**: 식별자를 받지만 내부적으로 인증된 사용자의 리소스만 필터링하는 경우.
- **간접 참조 테이블**: 사용자별 매핑 테이블을 통해 실제 리소스에 접근하는 경우.

## 후보 판정 제한

사용자가 제공한 식별자로 다른 사용자의 리소스에 접근할 수 있는 경로가 존재하는 경우 후보. 글로벌 미들웨어에서 소유권 검증이 수행되거나 데이터 접근 쿼리에 사용자 스코프가 포함되어 있으면 제외.
