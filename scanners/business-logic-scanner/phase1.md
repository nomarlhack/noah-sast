> ## 핵심 원칙: "코드의 의도와 실제 구현의 불일치를 찾는다"
>
> 이 스캐너는 grep 패턴에 의존하지 않는다. AI의 코드 의미 이해력으로 비즈니스 로직의 결함을 탐지한다. 기술적 취약점(XSS, SQLi 등)은 다른 스캐너가 담당하므로, 여기서는 **비즈니스 규칙 위반**만 다룬다.

## 분석 방법 (grep-less 접근)

이 스캐너는 패턴 인덱스 파일이 비어 있으므로, Sink-first(6-A) 대신 아래 접근법을 따른다.

### Step A: 상태 변경 API 전수 식별

프로젝트의 라우트 정의를 Read로 읽어 **상태를 변경하는 모든 엔드포인트**를 식별한다.

대상:
- POST/PUT/PATCH/DELETE 메서드의 모든 라우트
- GraphQL mutation
- WebSocket 메시지 핸들러 중 상태 변경이 있는 것

각 엔드포인트에 대해 핸들러 함수를 Read로 읽는다.

### Step B: 카테고리별 분석

식별된 상태 변경 API를 아래 7개 카테고리로 분석한다.

#### B-1: 가격/수량 조작 (Price/Quantity Tampering)

결제, 주문, 장바구니, 포인트, 쿠폰 관련 엔드포인트에서:
- 클라이언트가 보낸 가격/금액을 서버가 그대로 사용하는가? (서버사이드 가격 재계산 없음)
- 할인/쿠폰 적용 시 음수 금액, 100% 초과 할인, 중복 적용이 가능한가?
- 수량에 0, 음수, 소수점, 극대값을 보내면 어떻게 처리되는가?
- 포인트/마일리지 사용 시 보유량 초과 사용이 가능한가?

| 패턴 | 후보 조건 |
|---|---|
| `price = req.body.price` (서버 재계산 없음) | 후보 |
| `total = items.reduce(...)` (DB 가격 조회) | 제외 |
| `discount` 파라미터를 직접 받아 적용 | 후보 |
| 쿠폰 적용 후 서버사이드 총액 재계산 | 제외 |

#### B-2: 권한 상승 (Privilege Escalation)

역할/권한 변경, 관리자 기능 접근 관련 엔드포인트에서:
- 일반 사용자 토큰으로 관리자 전용 API에 접근 가능한가?
- role/permission 필드를 요청 바디에서 직접 변경 가능한가? (mass assignment)
- 다른 사용자의 역할을 변경할 수 있는가?
- 초대/가입 시 역할을 지정할 수 있는가?

**IDOR 스캐너와의 분담**:
- 본 스캐너 (`PRIV_ESCALATION`): **자기 자신**의 role/permissions/isAdmin 필드를 mass assignment로 변경하는 케이스 (자기 권한 상승)
- IDOR (`MASS_ASSIGNMENT`): `owner_id`/`tenant_id` 등 **소유관계** 필드를 변경해 다른 사용자 리소스를 탈취하는 케이스
- IDOR: **다른 사용자**의 role을 변경하는 cross-user 케이스 (`PUT /users/:id/role`)
- 한 라인에서 두 케이스가 동시 트리거되면 양쪽 후보로 등재하되 통합 보고서에서 (file:line) 키로 그룹핑한다.

| 패턴 | 후보 조건 |
|---|---|
| `User.update(id, req.body)` (role 필드 필터링 없음) | 후보 |
| DTO/allowlist로 변경 가능 필드 제한 | 제외 |
| 관리자 라우트에 role 체크 미들웨어 없음 | 후보 |
| `@PreAuthorize("hasRole('ADMIN')")` 적용됨 | 제외 |

#### B-3: 레이스 컨디션 (Race Condition / TOCTOU)

잔액 차감, 재고 감소, 쿠폰 사용, 투표, 좋아요 등 **원자성이 필요한 연산**에서:
- 잔액 확인 → 차감이 단일 트랜잭션/원자적 연산인가?
- `SELECT ... FOR UPDATE` 또는 낙관적 락(version column)이 있는가?
- Redis/메모리 카운터를 사용하는 경우 원자적 연산(INCR, DECR)인가?
- 쿠폰/초대코드 사용 시 중복 사용 방지 메커니즘이 있는가?

| 패턴 | 후보 조건 |
|---|---|
| `if (balance >= amount)` → `balance -= amount` (별도 쿼리) | 후보 |
| `UPDATE SET balance = balance - ? WHERE balance >= ?` (원자적) | 제외 |
| 쿠폰 사용 후 상태 변경이 별도 트랜잭션 | 후보 |
| `BEGIN` + `SELECT FOR UPDATE` + `UPDATE` + `COMMIT` | 제외 |

#### B-4: 기능 남용 (Feature Abuse / Rate Limiting)

SMS 발송, 이메일 발송, 파일 업로드, API 호출 등 **비용이 발생하는 기능**에서:
- Rate limiting이 적용되어 있는가? (미들웨어, 데코레이터, 인터셉터 등)
- Rate limiting이 우회 가능한가? (IP 기반만 → 프록시 우회, 세션 기반 → 재로그인)
- OTP/인증코드 발송에 횟수 제한이 있는가?
- 비밀번호 시도 횟수 제한이 있는가? (계정 잠금 또는 지연)

| 패턴 | 후보 조건 |
|---|---|
| SMS/이메일 발송 엔드포인트에 rate limiter 없음 | 후보 |
| `express-rate-limit` 미들웨어 적용됨 | 제외 (단, 설정값 확인) |
| OTP 발송 시 이전 OTP 무효화 없음 | 후보 |
| 로그인 실패 카운터 + 계정 잠금 구현 | 제외 |

#### B-5: 상태 머신 우회 (State Machine Bypass)

주문, 결제, 승인, 워크플로우 등 **단계적 프로세스**에서:
- 중간 단계를 건너뛰고 최종 단계 API를 직접 호출 가능한가?
- 결제 없이 주문 완료 처리가 가능한가?
- 취소된 주문을 다시 완료 상태로 변경 가능한가?
- 승인 대기 상태를 건너뛰고 승인 완료 상태로 직접 변경 가능한가?

| 패턴 | 후보 조건 |
|---|---|
| `order.status = req.body.status` (임의 상태 변경) | 후보 |
| 상태 전이 테이블/FSM으로 유효한 전이만 허용 | 제외 |
| 결제 확인 없이 `order.complete()` 호출 가능한 라우트 | 후보 |
| 이전 단계 완료 여부를 미들웨어/가드에서 검증 | 제외 |

#### B-6: 데이터 정합성 위반 (Data Integrity Violation)

비즈니스 규칙이 코드로 강제되지 않는 경우:
- 자기 자신에게 송금/팔로우/평가가 가능한가?
- 만료된 쿠폰/토큰을 사용 가능한가?
- 최대/최소 제한값을 우회 가능한가?
- 동일 리소스에 중복 작업(중복 결제, 중복 투표)이 가능한가?

#### B-7: 정보 노출 (Business Information Disclosure)

기술적 정보 노출(스택 트레이스 등)이 아닌 **비즈니스 데이터 노출**:
- 에러 메시지에서 다른 사용자의 존재/상태를 유추 가능한가? ("이미 등록된 이메일")
- 검색 API에서 권한 외 데이터가 결과에 포함되는가?
- 페이지네이션으로 전체 데이터 수가 노출되는가? (관리자 통계 등)
- 디버그/내부 필드가 API 응답에 포함되는가?

## 안전 패턴 카탈로그 (FP Guard)

- **서버사이드 가격 재계산**: 클라이언트 가격을 무시하고 DB/카탈로그에서 조회
- **트랜잭션 + 행 잠금**: `SELECT FOR UPDATE` 또는 낙관적 락(version column)
- **상태 전이 매트릭스/FSM**: 유효한 상태 전이만 허용하는 로직
- **Rate limiter 미들웨어**: 글로벌 또는 엔드포인트별 적용
- **DTO/Allowlist**: 변경 가능한 필드만 허용하는 입력 필터링
- **Idempotency key**: 중복 요청 방지 (결제 등)
- **비즈니스 규칙 서비스 계층**: 별도의 validation service에서 규칙 집행

## 후보 판정 의사결정

| 조건 | 판정 |
|---|---|
| 클라이언트 전송 가격을 서버가 재검증 없이 사용 | 후보 (라벨: `PRICE_TAMPER`) |
| mass assignment로 role/permission 변경 가능 | 후보 (라벨: `PRIV_ESCALATION`) |
| 잔액 차감이 비원자적 (읽기→확인→쓰기 분리) | 후보 (라벨: `RACE_CONDITION`) |
| 비용 발생 기능에 rate limiting 없음 | 후보 (라벨: `FEATURE_ABUSE`) |
| 단계 건너뛰기로 최종 상태 직접 도달 가능 | 후보 (라벨: `STATE_BYPASS`) |
| 상태 변경 엔드포인트(approve/cancel/transfer)에서 객체 소유권 검증 누락 | **IDOR 수직 상승 담당** — 본 스캐너 후보 아님 |
| 자기 송금/팔로우/평가, 만료 쿠폰 사용, 중복 결제 등 정합성 위반 가능 | 후보 (라벨: `DATA_INTEGRITY`) |
| 에러 메시지/페이지네이션/응답 메타로 비즈니스 정보 노출 | 후보 (라벨: `INFO_DISCLOSURE`) |
| 서버사이드 가격 재계산 + DB 조회 확인 | 제외 |
| 트랜잭션 + 행 잠금 확인 | 제외 |
| FSM/상태 전이 검증 확인 | 제외 |
| Rate limiter 미들웨어 적용 확인 | 제외 |

## 후보 판정 제한

비즈니스 로직의 의도적 설계를 취약점으로 오인하지 않는다. "이 동작이 비즈니스 요구사항에 의한 것인지, 구현 누락인지" 판단이 불가능한 경우 후보로 등록하되 전제조건을 명시한다.
