# Chain Analysis — 연계 분석 (Phase 2)

Phase 1에서 발견된 개별 취약점 후보를 입력받아, 후보 간 연계 관계를 분석하고 실제 공격 시나리오를 도출한다.

## 목적

Phase 1 스캐너는 각각의 취약점 유형(XSS, SSRF, IDOR 등)을 독립적으로 분석한다. 그러나 실제 공격은 여러 취약점을 조합하여 성립한다. 연계 분석은 **이미 발견된 후보들을 연결하여** 완전한 공격 시나리오를 구성한다.

## 입력

메인 에이전트가 연계 분석 에이전트에 전달하는 데이터:

1. **후보 마스터 목록**: `<PHASE1_RESULTS_DIR>/master-list.json`을 Read하여 모든 후보를 확인한다 (ID, 제목, 유형, 위치, Source→Sink 경로, 실제 URL 경로). 각 후보의 상세 분석(코드 스니펫, 검증 로직 등)은 `<PHASE1_RESULTS_DIR>/<scanner-name>.md`에서 읽는다.
2. **프로젝트 컨텍스트**: 기술 스택, 서비스 아키텍처, 인증 방식, 네트워크 구조
3. **이상 없음 요약**: Phase 1에서 안전으로 판정된 항목의 요약 (참고용)

> **작업 디렉토리에 여러 프로젝트가 존재하더라도, 모든 후보를 하나의 풀로 통합 분석한다.** 프로젝트 경계를 넘는 체인(예: hubrix의 SSRF → reacto의 데이터 노출)도 탐지 대상이다.

## 분석 프로세스

### Step 1: 전제조건 매트릭스 작성

각 후보에 대해 **트리거 전제조건**을 분석한다:

```
후보 ID: XSS-2
트리거 전제조건:
  1. [권한] 이용기관 자격으로 전자문서 발송 API 호출 가능
  2. [데이터] 수신자의 전화번호, 이름, 생년월일
  3. [네트워크] drowa-api 엔드포인트 접근 가능
```

모든 후보에 대해 전제조건을 나열한 뒤, **전제조건 유형별로 그룹화**한다:
- `[권한]`: 특정 API/서비스에 대한 인증/인가
- `[데이터]`: 특정 식별자(envelopeId, userId 등) 또는 개인정보
- `[네트워크]`: 특정 도메인/포트 접근
- `[환경]`: 특정 환경(sandbox/production) 한정 여부

### Step 2: 연계 관계 분석

후보 간 **"A가 B의 전제조건을 충족하는가?"** 관계를 분석한다.

연계 관계는 아래 조건을 모두 만족해야 인정한다:
(a) 후보 A의 출력 데이터/획득 권한/변경된 상태가 후보 B의 sink 입력으로
    코드 레벨에서 도달함
(b) 그 데이터 흐름이 파일경로:라인번호로 추적 가능함 (최소 2개 지점)
(c) 후보 B가 후보 A 없이는 실행 불가능, 또는 후보 A의 결과로 후보 B의
    트리거 조건 자체가 본질적으로 변함

조건 중 하나라도 미달 → "연계 없음(빈칸)". narrative 판정 금지.

**연계 매트릭스 형식:**

```
         | XSS-1 | XSS-2 | HOST-1 | HOST-2 | ... |
---------|-------|-------|--------|--------|-----|
XSS-1    |   -   |       |        |        |     |
XSS-2    |       |   -   |        |        |     |
HOST-1   |       |       |   -    |        |     |
HOST-2   |       |       |        |   -    |     |
...      |       |       |        |        |  -  |
```

셀 값:
- `→`: 행(A)의 출력이 열(B)의 sink 입력으로 코드 경로 입증됨
- `↔`: 양방향 데이터 흐름이 양쪽 모두 코드 경로 입증됨
- (빈칸): 연계 없음

`+` (영향 증폭) 셀은 사용 금지. "함께 있으면 더 나빠 보임"은 인과가
아니므로 빈칸으로 처리한다.

### Step 3: 체인 구성 규칙 자가 검사 (필수)

`sub-skills/chain-analysis/chain-construction-rules.md`를 Read 도구로 읽어,
정의된 5개 규칙(R1~R5)을 모든 잠재 체인에 적용한다.

검사 결과를 출력에 다음 표 형식으로 포함한다:

| 잠재 체인 | R1 | R2 | R3 | R4 | R5 | 판정 | 사유 |

판정이 "통과"인 체인만 Step 4에서 작성한다.

### Step 4: 공격 체인 구성

연계 매트릭스에서 도출된 관계를 기반으로, 시작점에서 최종 영향까지의 완전한 공격 시나리오를 구성한다.

**공격 체인 형식:**

```
## 공격 체인 #1: [체인 제목]

공격자 프로필: [누가 이 공격을 수행할 수 있는가]
최종 영향: [공격 성공 시 결과]

Step 1: [후보 ID] — [설명]
  → [후보 A를 악용하여 얻는 것]

Step 2: [후보 ID] — [설명]
  → [Step 1의 결과를 이용하여 후보 B를 트리거]

Step 3: [후보 ID 또는 최종 영향] — [설명]
  → [최종 영향]

관련 후보: [사용된 Phase 1 후보 ID 목록]
```

**체인 분류:**
- **완전 체인**: 모든 단계가 Phase 1 후보로 구성됨. 동적 테스트에서 end-to-end 검증 가능.
- **부분 체인**: 일부 단계에 외부 전제조건(예: 내부망 접근)이 포함됨. 해당 전제조건 하에서 검증 가능.

### Step 5: 독립 후보 정리

어떤 체인에도 포함되지 않은 후보를 별도로 정리한다. 후보별 체인 미구성 사유만 기재한다. 단독 위험도는 각 취약점 상세 섹션에 이미 기술되어 있으므로 여기서 중복하지 않는다.

```
| 후보 | 체인 미구성 사유 |
|------|----------------|
| [후보 ID] | [이 후보가 다른 후보와 연계되지 않는 구체적 이유] |
```

## 출력

연계 분석 에이전트는 다음을 반환한다:

1. **전제조건 매트릭스**: 후보별 전제조건 테이블
2. **연계 매트릭스**: 후보 간 연계 관계 테이블 (`→`/`↔`/빈칸만 사용, `+` 금지)
3. **체인 구성 규칙 검사 결과**: 잠재 체인 N건 중 폐기 M건, 통과 L건. 폐기된 각 체인은 사유 태그(R1~R5) 명시.
4. **공격 체인 목록**: Step 3 통과한 것만. 없으면 "없음".
5. **독립 후보 테이블**: 후보별 체인 미구성 사유 (폐기된 체인의 관련 후보 모두 포함)
6. **Phase 3 테스트 목표**: 동적 테스트에서 검증할 체인/항목 목록 (체인 단위 우선)

## 에이전트 실행 지침

### 분석 원칙

1. **발견된 후보만 사용한다.** 새로운 취약점을 탐색하지 않는다. Phase 1 결과를 입력으로만 사용한다.
2. **코드 근거를 명시한다.** 연계 관계를 주장할 때, 해당 데이터 흐름의 코드 경로(파일경로:라인번호)를 제시한다. 데이터 흐름을 확인하기 위해 코드를 Read하는 것은 허용한다.
3. **추측 체인을 만들지 않는다.** "A의 출력이 B의 입력이 될 수 있다"는 코드 레벨에서 증명 가능해야 한다.
4. **공격자 프로필을 명확히 한다.** 각 체인이 누구에 의해 실행 가능한지(외부 공격자, 내부자, 이용기관 등) 명시한다.
5. **체인 0건은 정상 결과다.** 후보가 많아도 진정한 인과 흐름이 없으면 체인 0건으로 결론짓는다. 억지 체인 1건보다 무체인 결론이 우월하다.
6. **체인 작성 전 Step 3 체인 구성 규칙 검사를 반드시 통과한다.** 검사 결과는 출력에 항상 포함한다.

### 주의사항

- Phase 1에서 "안전"으로 판정된 항목을 재분석하지 않는다.
- 체인이 하나도 구성되지 않을 수 있다. 그 경우 "연계 관계 없음"으로 결론짓고 독립 후보 목록만 반환한다.
