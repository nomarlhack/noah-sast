# scan-report-review 공통 판정 원칙

3모드(`phase1-review`, `phase2-review`, `review의 단일 진실 원천. 각 모드 파일이 진입 직후 Read하여 적용한다. 모드 파일에 복제 금지.

---

## 1. Source 도달성 판정

Sink 인자가 외부 행위자(공격자·타 사용자·외부 시스템·외부 쓰기 가능 저장소)에 의해 제어 가능한지 판정한다.

### 역추적 절차

1. Sink 호출부의 인자 변수를 선택
2. 같은 함수 내 최초 대입 지점 Read로 확인
3. 함수 파라미터에서 시작했으면 호출자를 Grep하여 값을 확인 (**최대 3단계 재귀**)

### 판정 원칙

**의미 기반** 판정. 패턴 목록·enum에 의존하지 않는다.

**✗ 폐기**: 최초 대입 지점이 외부 영향과 **인과적으로 분리**되어 프로그램 내부에서 결정론적으로 산출되는 값 (리터럴, 내부 시계/난수, 모듈 내 계산 결과 등).

**✓ 유지**: 최초 대입 지점이 **외부 행위자에 의해 직접·간접 제어 가능한 채널**. 프로토콜·스택 불문, "외부에서 결정되는가"가 유일 기준.

**판단 방법**: *"이 값을 외부 행위자가 의도적으로 다르게 만들 방법이 코드 외부에 존재하는가?"*
- 존재 → ✓ 유지
- 존재하지 않음을 코드로 입증 가능 → ✗ 폐기
- 입증 불가 → 불명확 케이스로 처리

### 혼합 케이스

Sink가 여러 호출부를 가지며 일부는 ✓ 일부는 ✗이면, 각 호출 경로를 분리해 ✓ 경로만 후보로 유지하고 호출 경로 정보를 후보 기술에 포함한다.

### 불명확 케이스 — 보수적 유지

확신 없는 폐기보다 안전한 판정이다. 다음 중 하나라도 해당하면 후보를 폐기하지 말고 유지하며 "Source 도달성 불명확"을 후보 기술에 명시한다.

- 외부 제어 가능성을 확신할 수 없지만 배제도 못 하는 경우
- 역추적이 리플렉션·동적 dispatch·IoC 컨테이너 주입·메타프로그래밍 등으로 3단계 내 확정 불가
- 스택/프레임워크가 낯설어 판정 자체가 불확실

## 2. 부재 주장 검증

"~가 없다", "~하지 않는다" 주장을 모두 검증한다 (스킴만 검증하면서 "검증 없음"으로 기술하는 등 놓치는 경우 방지).

### 필수 절차

1. 해당 함수/메서드의 전후 컨텍스트를 **최소 ±30줄** Read
2. 같은 클래스/파일 내 다른 메서드에서 검증하는지 확인
3. **호출자 체인**을 거슬러 올라가며 상위 레이어의 검증 확인
4. **프레임워크 내장 방어** 확인 (Spring Security, React 자동 이스케이프, OWASP ESAPI 등 — 버전별 차이 주의)
5. 설정 파일(`SecurityConfig`, `WebMvcConfig` 등)의 전역 필터 확인

### 판정 처리

실제로 검증 로직이 존재하면:

- 검증이 완전하여 취약점이 아닌 경우 → 후보에서 제거 (모드별 절차에 따라 재분류)
- 검증이 존재하지만 우회 가능한 경우 → 검증 로직을 결과에 명시하되 우회 방법 기술 후 후보 유지
- 검증이 존재하지만 불완전한 경우 (예: 스킴만 검증, 호스트 미검증) → 내용을 정확하게 수정

## 3. 반환 형식 필수 규칙

### `Source 도달성:` 라인

- 후보/확인됨 항목에서 **생략 불가**. 이 라인이 없는 항목을 포함한 반환은 거부된다.
- 이상 없음 항목은 `— 해당 없음` 사용.

### 판정별 근거 동반 의무

| 판정 | 필수 근거 | 예시 |
|------|----------|------|
| `✗ 폐기` | 폐기 근거 명시 | `✗ 폐기 (인자 X가 static final 상수 "abc"로 대입, line 42)` |
| `✓ 유지` | 실제 역추적한 Source 지점 | `✓ 유지 (source: request.getParameter("url"), UserController.java:42)` |
| `? 불명확` | §1의 불명확 케이스 조건 명시 | `? 불명확 (역추적 3단계 내 확정 불가: IoC 컨테이너 주입)` |

근거 없는 판정은 무효이며 거부된다.

### `[혼합]` 포맷

Sink가 여러 호출 경로를 가지며 판정이 경로마다 다를 때:

```
Source 도달성 [혼합]:
  - 경로 1: ✓ 유지 (source: request.getParameter("q"), MixedCaller.java:16 → write():12)
  - 경로 2: ✗ 폐기 (인자가 HTML 리터럴 "<b>Fixed banner</b>", MixedCaller.java:21)
```

호출하는 측의 해석:
- ✓ 경로가 1개 이상이면 후보 유지 (후보 기술을 ✓ 경로만으로 한정) → 집계는 **유지_count**에만 포함
- 모든 경로가 ✗이면 폐기 → 집계는 **폐기_count**에만 포함
- 한 후보가 두 버킷에 동시 집계되지 않음 (혼합 케이스는 ✓ 우세 규칙)
