# MODE GUARD: 이 파일은 mode=report-review 전용

**[STOP]** 진입 프롬프트에 `MODE=report-review`가 명시되지 않았다면 즉시 종료하고 "모드 불일치"를 보고한다.

**다른 모드라면**:
- mode=phase1-review → `phase1-review.md`
- mode=phase2-review → `phase2-review.md`

**진입 전 필수 Read**: `_principles.md`, `_contracts.md`

**역할**: 조립된 보고서 MD 본문의 **설명 품질 개선**. 스니펫·경로·흐름 보정, POC 명령어 수정, 중복 항목 통합, 부족한 원인 분석·권장 조치 보강.

**입력 허용**:
- 조립된 `noah-sast-report.md` (주요 수정 대상)
- `<PHASE1_RESULTS_DIR>/master-list.json` (Read)
- `<PHASE1_RESULTS_DIR>/evaluation/<scanner>-eval.md` (Read)
- `<PHASE1_RESULTS_DIR>/<scanner>-phase2.md` (Read)
- `<PHASE1_RESULTS_DIR>/chain-analysis.md` (Read)
- 프로젝트 소스코드 (Read)

**수정 허용 영역** (보고서 MD 본문 한정):
- `#### 원인 분석` 섹션 본문
- `#### 재현 방법 및 POC` 섹션의 Step 설명·curl 명령
- `#### 권장 조치` 섹션 본문
- `**위치**:` 필드의 파일 경로
- `**Source**:`·`**Sink**:` 필드 텍스트
- 코드 블록 스니펫
- 연계 시나리오 섹션의 Step 설명

**수정 금지 영역**:
- `**ID**:`·`**유형**:`·`**상태**:`·`**미확인 사유**:` 필드 값 (status·tag writer는 phase2-review)
- 총괄 요약 건수·취약점 요약 테이블 구조
- `## 안전 판정 항목` 섹션 (assemble_report.py 자동 생성)
- `## 미적용 스캐너 목록` 테이블
- `#### N. 제목` 헤딩 개수·구조

**금지**:
- master-list.json·eval MD·Phase 2 manifest 쓰기
- **판정 전환** (safe ↔ candidate ↔ confirmed) 시도 — 본문 설명만 보강하고 `**상태**:` 필드 불변
- Phase 1 원본 `<scanner>.md` / `ai-discovery.md` 직접 참조 (`_contracts.md §6` C1 lint)

## Step 1: 보고서 파싱

## Step 2: 품질 검증 (병렬 서브에이전트)

항목을 그룹당 최대 5개로 분할. 각 서브에이전트가 체크리스트 적용.

### 체크리스트 10항목 (수정 범위)

| # | 항목 | 수정 대상 |
|---|------|---------|
| 1 | 파일 존재 | `**위치**:` 경로가 실재하지 않으면 실제 경로로 교정 |
| 2 | 코드 스니펫 | 인용 코드가 실제 라인에 없으면 실제 코드로 교체 (±5줄 허용) |
| 3 | 데이터 흐름 | Source→Sink 설명이 실제 호출과 다르면 원인 분석 재작성 |
| 4 | 부재 주장 | `_principles.md §2` 적용. 검증 로직이 실재하면 원인 분석에 "부분 검증 존재, 우회 가능 여부 기술" 보강 (판정 유지) |
| 5 | 검증 로직 누락 | 보고서가 언급하지 않은 검증 로직을 원인 분석에 추가 |
| 6 | 이상 없음 근거 | 안전 판정 근거 설명 보강 (`**상태**:` 유지) |
| 7 | 복수 요소 커버리지 | 복수 설정값/파일 결함 시 누락 요소를 본문·POC에 추가 |
| 8 | 동일 file:line 다층 관점 | 중복 섹션을 하나로 통합 (헤딩 수 줄어들면 총괄 요약 테이블도 함께 갱신, assemble_report.py 파싱 형식 보존) |
| 9 | Source 도달성 | `_principles.md §1` 적용. 도달 경로 기술을 원인 분석에 보강 (판정 유지) |
| 10 | POC 검증 | curl 엔드포인트·HTTP 메서드·파라미터명이 실제 라우트와 일치하도록 교정 |

**판정 전환 발견 시**: review는 `**상태**:` 필드를 건드리지 않는다. 본문 설명 품질만 개선하고, 판정 자체의 정확성은 phase1-review/phase2-review의 책임 범위로 둔다.

### 서브에이전트 반환 형식

```
=== 후보: [ID] [제목] ===
Source 도달성: ✗ 폐기 (근거) / ✓ 유지 (source: <표현식>, <파일>:<라인>) / ? 불명확 (조건) / — 해당 없음
체크리스트 1-10: [항목별 ✓ / ✗ / 해당없음]
판정: ✓ 정확 / ⚠ 본문 보강 필요 / ✗ 심각 오류
[⚠ 또는 ✗인 경우]
문제: [체크리스트 번호 + 내용]
실제: [소스/데이터에서 확인한 실제]
수정안: [본문에 반영할 교정 내용]
```

혼합 케이스는 `_principles.md §3 [혼합]` 포맷.

## Step 3: Source 도달성 반환 검증

1. 반환에 `Source 도달성:` 라인이 없는 항목 → 재검증 서브에이전트 디스패치.
2. 누락 재현 시 메인 에이전트가 직접 역추적.
3. **Spot-check**: ✓ 유지 항목에서 최대 3개 샘플링. `(source: <표현식>, <파일>:<라인>)`을 Read로 확인. 실제 위치에 없으면 **허위 증거**로 간주, 해당 서브에이전트 반환 전부 재검증.

## Step 4: MD 본문 수정

서브에이전트 반환 중 `⚠ 본문 보강 필요` 또는 `✗ 심각 오류` 항목을 Edit로 수정.

### 수정 유형별 처리

| 유형 | 조치 |
|------|------|
| 스니펫/라인 번호 오류 | Edit 도구로 해당 블록만 교체 |
| Source→Sink 흐름 오류 | `#### 원인 분석` 섹션 재작성 |
| 검증 로직 존재 (부재 주장 거짓) | `#### 원인 분석`에 검증 로직 기술 추가, "우회 가능 여부" 보강 |
| 복수 요소 누락 | 본문·POC에 누락 요소 추가 |
| 동일 file:line 중복 | 두 섹션을 하나로 통합, 총괄 요약·취약점 요약 테이블 건수 재계산 |
| POC 오류 | curl 명령어의 URL·메서드·파라미터 교정 |
| Source 도달성 보강 필요 | 원인 분석에 실제 도달 경로 기술 추가 (판정 유지) |

### 형식 유지 [필수]

Edit 도구 사용 시 아래를 모두 지킨다:
- `#### N. 제목` 헤딩 수를 건드리면 총괄 요약·취약점 요약 테이블도 동시에 갱신 (단, 구조 유지: assemble_report.py가 재조립 없이 파싱 가능해야 함)
- `**ID**:`·`**유형**:`·`**상태**:`·`**미확인 사유**:` 필드 값 불변
- 코드 블록 펜스(```) 정합성 유지
- `_contracts.md §9` 내부 용어(mode명, DISCARD/OVERRIDE/CONFIRM, `assert_*.py` 등)를 헤딩·소제목에 노출 금지

## 자가 검증 (반환 전 필수)

1. **커버리지 100%** — 후보/확인됨·체인·이상 없음 `검증 N == 전체 M`.
2. **Source 도달성** — 모든 후보에 라인 + 근거 명시.
3. **부재 주장** — `_principles.md §2` 적용.
4. **체인** — M_chains > 0이면 R1~R5 재적용 (본문 설명 보강 범위 내).
5. **Writer 권한** — master-list.json·eval MD·Phase 2 manifest 쓰기 없음.
6. **판정 불변** — `**상태**:` 필드 전부 원본 유지.
7. **형식 보존** — `#### N. 제목` 헤딩 수 변동이 있으면 총괄 요약 테이블도 동기화, assemble_report.py 파싱 형식 파괴 없음.
8. **독자 레이어** — 보고서 MD에 `_contracts.md §9` 내부 규약 용어 노출 없음.
