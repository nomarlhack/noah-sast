# 독자 레이어 용어 노출 lint (`lint_reader_layer.py`)

보고서 MD/HTML의 **독자 가시 영역**에 스킬 내부 규약 용어/내부 메타 정보가 노출되지 않았는지 자동 검사하는 lint 도구.

## 왜 필요한가

보고서는 **개발·운영 리더가 읽고 조치를 결정하는 문서**다. 스킬 내부 파이프라인 서술(grep → Phase 1 → evaluate_phase1 → 조립), 판정 라벨(DISCARD/OVERRIDE/CONFIRM), 스크립트명, checklist 섹션 번호(§N) 같은 **내부 운영 용어**가 독자에게 그대로 노출되면:

- 독자는 용어의 의미를 알지 못해 보고서 이해가 막힌다
- "프로세스가 어떻게 돌아가는지" 같은 내부 메타 정보가 판단 근거와 뒤섞인다
- 과거 보고서 리뷰에서 반복적으로 지적된 결함이다

이 lint는 **메인 에이전트/서브에이전트가 실수로 이런 정보를 독자 가시 영역에 쓸 경우 자동으로 차단**한다.

## 검사 범위 — 모두 화이트리스트 + 블랙리스트 backup

### 1. 헤딩 (화이트리스트 + 블랙리스트 backup)

**대상**: 모든 MD/HTML 헤딩. 코드블록 내부 `#` 주석은 헤딩이 아니므로 스킵.

**1차 (화이트리스트)**: `vuln-format.md` "보고서 섹션 구조" 섹션에서 **고정 헤딩 집합 + 템플릿 헤딩 정규식**을 파싱하여 허용 스펙으로 사용. 스펙에 정의되지 않은 헤딩은 차단.

**2차 (블랙리스트 backup)**: 화이트리스트 통과한 헤딩도 `BANNED_PATTERNS` 검사. `{VULN_TITLE}` 같은 자유 텍스트 템플릿에 내부 용어(`DISCARD` 등)가 끼어드는 것을 막는 이중 안전망.

```markdown
## DISCARD 판정 후보       ← 차단 (스펙 외 헤딩)
## §12 검증 결과           ← 차단 (스펙 외 헤딩)
## 전체 요약               ← 차단 (스펙 외 — 정식은 '총괄 요약')
## 폐기 판정 항목          ← 차단 (스펙 외 — 한글 변형이어도 차단)
## 안전 판정 항목          ← 허용 (고정 스펙)
### XSS Scanner           ← 허용 (템플릿 `{SCANNER_NAME} Scanner`)
### XSS 분석기             ← 차단 (템플릿 미매칭)
#### 1. 자유 제목           ← 허용 (템플릿 `{N}. {VULN_TITLE}`)
#### 1. DISCARD 포함 제목   ← 차단 (템플릿 통과 but BANNED backup)
```

#### MD/HTML 레벨 매핑

보고서 렌더링 과정에서 MD의 `#### N. 제목`(h4)이 HTML에서 `<h3>`로 승격된다 — `md_to_html.py`의 `details.vuln-block>summary h3` CSS 재사용 구조 때문. lint는 이 특이사항을 알고 **HTML vuln-block 내부 `<h3>`를 스킵**한다. 동일 내용이 MD lint에서 이미 검사되므로 중복 불필요.

### 2. 개요 필드명 (화이트리스트 방식 — `vuln-format.md` 단일 출처)

**대상**: 보고서 최상단 개요 블록 내 `**필드명**:` 형태 라인의 **필드명 부분만** 검사. 값은 자유.

**방식**: `vuln-format.md`의 `## 통합 보고서 구조` 섹션 내 첫 번째 ```markdown 코드 블록을 **단일 진실 원천**으로 파싱하여 허용 필드 집합을 자동 추출. 실제 보고서의 개요 필드 이름이 이 집합에 없으면 차단.

**블랙리스트 없음**. 단어 변형(`파이프라인` → `파이프 구성` → `스캔 단계` → ...)에도 무관하게, 스펙에 등록되지 않은 필드는 모두 차단된다.

개요 블록 식별:
- 첫 `# 제목` 헤딩 직후 ~ 첫 `---` 또는 다음 `##` 헤딩 이전
- `## 개요` 서브 헤딩이 있으면 그 이후를 개요 컨텐츠 시작으로 간주

예시 (스펙이 `대상/스캔 일시/스캔 방식/테스트 환경/스택` 5개로 정의된 경우):

```markdown
# 통합 취약점 스캔 보고서

## 개요

**대상**: kakao-developers                 ← 허용 (스펙에 있음)
**스캔 일시**: 2026-04-18                   ← 허용
**스캔 방식**: 소스코드 분석 + 동적 테스트   ← 허용
**테스트 환경**: sandbox-developers...       ← 허용
**스택**: Node.js / nginx                   ← 허용
**파이프라인**: grep → Phase 1 → 조립        ← 차단 (스펙 외)
**파이프 구성**: ...                        ← 차단 (단어 바꿔도)
**스캔 단계**: 1→2→3                        ← 차단 (단어 바꿔도)
**감사 ID**: AUDIT-2026-001                 ← 차단 (합법 후보라도 스펙 외면 차단)
```

## 헤딩 블랙리스트 (`BANNED_PATTERNS`) — backup 안전망

헤딩 화이트리스트를 통과했어도 이 정규식 목록에 매칭되면 차단. 자유 텍스트 템플릿(`{VULN_TITLE}` 등)에 내부 용어가 섞이는 경우 대비. 개요 필드 검사에는 사용하지 않음.

| 토큰 | 설명 |
|------|------|
| `§\s*\d+` | checklist.md 섹션 번호 (§N) |
| `\bmode\s*=\s*(evaluate_phase1\|evaluate\|review)\b` | mode명 |
| `\bevaluate_phase1\b` | evaluate_phase1 |
| `\b(DISCARD\|OVERRIDE\|CONFIRM)\b` | 내부 판정 라벨 |
| `Source\s*도달성` | 내부 판정 용어 |
| `실질\s*영향\s*반증` | 내부 판정 용어 |
| `\b[a-z0-9_-]+\.py\b` | 스크립트 파일명 |
| `phase1_(validated\|discarded_reason\|eval_state)` | master-list 내부 필드명 |
| `safe_category` | 내부 분류 필드명 |

### 허용되는 경우 (헤딩)

- **본문/테이블 근거 서술**에서 풀이 형태로 사용: 허용 (예: "evaluate 평가에서 차단으로 판정됨"은 테이블 cell에서 OK)
- **개요 필드의 값** 부분: 허용 (필드명만 검사)
- **코드 블록·인용 블록**: 헤딩이 아니면 스킵

## 사용법

```bash
python3 tools/lint_reader_layer.py <report.md> [<report.html>]
```

예시:

```bash
# MD만 검사
python3 tools/lint_reader_layer.py noah-sast-report.md

# MD + HTML 모두 검사 (권장)
python3 tools/lint_reader_layer.py noah-sast-report.md noah-sast-report.html
```

## Exit Code

| exit | 의미 |
|------|------|
| `0` | pass — 위반 없음 |
| `5` | lint 위반 — 헤딩 금지 토큰 또는 개요 스펙 외 필드 |
| `1` | CLI 인자 오류 |

보고서 생성 파이프라인(scan-report `Step 4`)에서 lint exit 5가 발생하면 수정 후 재조립 필요.

## 위반 발생 시 대응

### 개요 필드명 위반
두 가지 경로 중 하나 선택:

1. **해당 필드를 삭제**하거나 다른 허용 필드의 값에 통합 (예: "인증" 정보를 `**스택**:` 값에 포함).
2. **필드가 진짜로 필요한 경우 `vuln-format.md` 스펙을 먼저 갱신**:
   - `## 통합 보고서 구조` 섹션 내 ```markdown 코드 블록에 `**<새 필드명>**: [값 예시]` 줄 추가.
   - lint는 다음 실행 시 자동으로 새 필드를 허용 집합에 포함.
   - 이 경로가 "근본 변경 경로" — 스펙이 곧 lint 규약이므로 **독자 레이어 가시 정보의 유일한 정식 등록 방법**.

### 헤딩 위반
헤딩 텍스트를 독자 친화적 표현으로 교체. 필요한 정보는 본문 근거 테이블에 풀이 형태로 이관.

수정 후 `assemble_report.py` 재실행 → lint 재실행하여 통과 확인.

## 확장/유지보수

### 개요 필드 확장 (권장 경로)

`vuln-format.md`의 "통합 보고서 구조" 코드 블록 수정만으로 lint가 자동 반영. 코드 변경 불필요.

```markdown
## 통합 보고서 구조

```markdown
# 통합 취약점 스캔 보고서

**대상**: [프로젝트명]
**스캔 일시**: [날짜]
**스캔 방식**: 소스코드 분석 + 동적 테스트
**테스트 환경**: [sandbox 도메인]
**스택**: [프레임워크/언어/DB 등]
**감사 ID**: [예: AUDIT-2026-001]   ← 이 줄 하나 추가만으로 lint 허용
```
```

### 헤딩 금지 토큰 추가

새 내부 용어가 도입되어 헤딩에서 차단해야 할 경우:

1. `tools/lint_reader_layer.py`의 `BANNED_PATTERNS` 리스트에 `(정규식, 설명)` 튜플 추가
2. 회귀 테스트: 기존 보고서에 lint 실행하여 false positive 확인

## 설계 원칙

- **개요 / 헤딩 모두 화이트리스트** (`vuln-format.md` 단일 출처). 단어 변형·오타·임의 삽입에 일관되게 강건.
- **블랙리스트 `BANNED_PATTERNS`**는 화이트리스트 통과 후 2차 안전망으로만 사용 (헤딩 한정). 자유 텍스트 템플릿에 내부 용어가 끼어드는 엣지를 방어.
- 스펙 변경 = lint 규약 변경. 코드 수정 없이 문서(`vuln-format.md`)만 갱신하여 일관성 유지.
- 코드블록 내부의 `#` 주석·`**필드명**:` 형태 라인은 검사 제외 (MD는 전처리로 공백화, HTML은 해당 구조 없음).

## 관련 자산

- `sub-skills/scan-report/vuln-format.md` — **개요 허용 필드 스펙**(단일 진실 원천) + 헤딩 검사 규약
- `sub-skills/scan-report/validate_report.py` — 정량/구조 검증 (POC 개수, ID 필드, URL 일관성)
- `sub-skills/scan-report/validate_links.py` — HTML 앵커 링크 검증
- `docs/review-modes.md` — evaluate/review 모드 상세 가이드

## 확장 이력

- **v1**: 헤딩(h1~h6)만 검사. 9개 금지 토큰.
- **v2**: 개요 섹션 `**필드명**:` 검사 범위 추가. (초기 구현은 블랙리스트 3개 추가)
- **v3**: 개요 검사를 `vuln-format.md` 단일 출처 화이트리스트로 재설계. 블랙리스트 3개 제거.
- **v4 (현재)**: 헤딩 검사도 화이트리스트 + 블랙리스트 backup 구조로 전환. 고정 헤딩 20개 + 템플릿 7종을 vuln-format.md "보고서 섹션 구조" 섹션에서 자동 파싱. 코드블록 내부 `#` 주석 스킵, HTML vuln-block 내부 h3 스킵 로직 추가. 단어 변형·오타·임의 섹션 삽입 모두 차단.
